# STATE — memória do projeto

## Projeto
Reconstrução do SaaS **Clínica Experts** (gestão de clínicas, pt-BR) a partir de 36 specs
em `docs/paginas/` + 58 screenshots em `images/`. App web em `web/`.

## Decisões
- **D1** — Stack UI: Next.js 16 (App Router, Turbopack) + Tailwind v4 + componentes shadcn-style hand-rolled + Recharts + lucide-react. Tailwind v4 usa `@theme` em CSS (sem tailwind.config). shadcn CLI evitado (instável no Next 16); primitives feitas à mão em `src/components/ui/`.
- **D2** — Dados: mock estático em `src/lib/mock.ts`, valores exatos dos screenshots.
- **D3** — Cor de marca roxo `#7c3aed` (token `--brand`). Sem dark mode nesta fase.
- **D4** — Charts são Client Components (`"use client"`); páginas seguem Server Components.
- **D5** — Rotas simplificadas: `/`, `/agenda`, `/pacientes`, `/financeiro` (specs reais usam `/clinica/...`).
- ~~**D6** — Backend Go (net/http + pgx, auth argon2id+sessions, RLS via `SET LOCAL app.clinica_id`)~~ — **RE-REVOGADA em 2026-06-29 por D7.** Backend Go sai por inteiro; frontend fala direto com Supabase. O Go foi implementado e validado contra Supabase real (ver Concluído), mas a operação de 2 deploys (Vercel + host Go) não compensa quando o Postgres já é Supabase gerenciado. Modelo de domínio (21 tabelas, `clinica_id`, mapa JSONB, ledgers) **preservado**; muda só onde a lógica roda.
- **D7** — Arquitetura: **Supabase direto, sem backend** (3º giro: Drizzle → Go → Supabase direto). Frontend Next.js fala com Supabase via `@supabase/ssr` (anon key). **Supabase Auth nativo** (`usuarios` = perfil ligado a `auth.users`; `sessions`/argon2id eliminados). **RLS deriva tenant de `auth.uid()`** via função `auth_clinica_id()` `security definer` (substitui `SET LOCAL app.clinica_id`). **CRUD via PostgREST** (interface `Collection<T>` intacta). **Lógica pesada (dashboard/financeiro/agenda) computa no frontend** (re-porta `internal/domain` → libs puras); **EXCETO estoque delta**, que vira **RPC atômico `FOR UPDATE`**. Register via RPC `criar_clinica` (`SECURITY DEFINER`, idempotente). Deploy único Vercel (root `web`). Spec canônica: `.specs/features/supabase-direct/` (spec+design+tasks). Migração **completa de uma vez**; `backend/` removido do fluxo (deleção física = limpeza pós-smoke).
